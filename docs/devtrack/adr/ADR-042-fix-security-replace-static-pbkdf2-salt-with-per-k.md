# ADR — fix(security): replace static PBKDF2 salt with per-key random salt

> **Date:** 2026-07-04 | **PR:** #3060 | **Status:** Accepted

## Context

The previous API key authentication mechanism used a hardcoded static salt (`"sahidawa-api-key-v1"`) with PBKDF2 to hash API keys. This static salt made the system vulnerable to precomputation attacks (e.g., rainbow tables) if the database was compromised. 

Furthermore, because the database was queried using the computed hash (`.eq("key_hash", keyHash)`), the system had to hash the incoming key *before* querying. This design made it impossible to use unique, per-key salts because the salt could not be retrieved from the database without first identifying the record. The comparison of hashes was also performed using standard string comparison, exposing the system to timing attacks.

## Decision

We modified the API key architecture to support per-key random salts and constant-time verification:

1. **Database Schema Update:** Added a `key_salt` column to the `api_keys` table via a Supabase migration to store a unique, hex-encoded 32-byte random salt for each key.
2. **API Key Format Change:** Changed the API key format from a single raw token to a composite `id.secret` format.
3. **Lookup and Verification Flow:**
   - The middleware splits the incoming API key into `keyId` and `secret`.
   - It queries the database using the `id` to retrieve the `key_hash` and `key_salt`.
   - It computes the PBKDF2 hash of the incoming `secret` using the retrieved `key_salt` (100,000 iterations, SHA-512, 64-byte length) asynchronously via libuv's thread pool.
   - It performs a constant-time comparison of the computed hash and the stored hash using `crypto.timingSafeEqual`.

## Alternatives Considered

| Alternative | Why Rejected |
|---|---|
| **In-Memory Cache Lookup** | We considered keeping a single-token API key format and using an in-memory cache (e.g., Redis) to map key prefixes to salts. This was rejected because it introduces external operational dependencies, cache invalidation complexity, and increased infrastructure costs for rural deployment scenarios. |
| **Bcrypt / Argon2id Hashing** | We considered migrating from PBKDF2 to Argon2id or bcrypt. This was rejected because PBKDF2 is natively supported in Node.js without native binary compilation issues, and running it asynchronously with 100k iterations provides sufficient cryptographic strength without blocking the event loop. |

## Consequences

**Positive:**
- Eliminated the risk of precomputation and rainbow table attacks against the API key database.
- Mitigated timing attacks on API key verification by utilizing `crypto.timingSafeEqual`.
- Improved query performance by searching the database using a primary/indexed `id` rather than a long hash string.

**Trade-offs:**
- **Breaking Change:** All existing API keys were invalidated and must be regenerated by clients, as the raw secrets were never stored and cannot be migrated to the new salted schema.
- **Database Roundtrip:** The database must now be queried before the cryptographic hash can be verified, which could theoretically expose the system to database connection exhaustion if flooded with invalid IDs (mitigated by standard rate limiting).
