# PR #3085 — refactor(api): Use centralised cacheMiddleware in pharmacy routes

> **Merged:** 2026-07-04 | **Author:** @Himanshujha7 | **Area:** Backend | **Impact Score:** 9 | **Closes:** #2852

## What Changed

We refactored our pharmacy routes to eliminate redundant caching logic by removing the hardcoded `GEOSPATIAL_CACHE_CONTROL` constant and the `setGeospatialCacheHeaders` helper function from `apps/api/src/routes/pharmacies.ts`. Instead, we fully consolidated cache-header handling under our centralized `cacheMiddleware`. Additionally, we updated our integration tests in `apps/api/tests/pharmacies.test.ts` to dynamically derive the expected `Cache-Control` header value from the middleware itself rather than asserting against a hardcoded string literal.

## The Problem Being Solved

Prior to this PR, our geospatial pharmacy endpoints (`/nearest` and `/in-bounds`) had duplicate caching mechanisms. Although `cacheMiddleware(300, 600)` was already configured at the router level, the route handlers themselves manually invoked `setGeospatialCacheHeaders(res)` inside their response branches. This resulted in dead code and bypassed the single-responsibility principle of our middleware pipeline. 

Furthermore, our test suite asserted against a hardcoded string literal (`GEOSPATIAL_CACHE_CONTROL = "public, max-age=300, s-maxage=300, stale-while-revalidate=600"`). If the centralized `cacheMiddleware` implementation or formatting ever changed, the tests would silently drift out of sync, leading to false positives or unexpected test failures during future updates.

## Implementation Details

### Route Refactoring
In `apps/api/src/routes/pharmacies.ts`, we removed the local constant and helper function:
```typescript
// REMOVED
const GEOSPATIAL_CACHE_CONTROL = "public, max-age=300, s-maxage=300, stale-while-revalidate=600";

const setGeospatialCacheHeaders = (res: Response) => {
    res.setHeader("Cache-Control", GEOSPATIAL_CACHE_CONTROL);
};
```
We then removed all invocations of `setGeospatialCacheHeaders(res)` within the `router.get("/nearest", ...)` and `router.get("/in-bounds", ...)` response blocks. Because the router already registers `cacheMiddleware(300, 600)` upstream, the Express response lifecycle automatically handles setting these headers before the payload is sent to the client.

### Test Suite Refactoring
In `apps/api/tests/pharmacies.test.ts`, we imported the centralized `cacheMiddleware` and introduced a dynamic helper function to generate the expected header value:
```typescript
import { cacheMiddleware } from "../src/middleware/cache";

function cacheControlFor(durationSeconds: number, staleWhileRevalidateSeconds: number): string {
    let headerValue = "";
    const fakeRes = { setHeader: (name: string, value: string) => (headerValue = value) } as never;
    cacheMiddleware(durationSeconds, staleWhileRevalidateSeconds)({} as never, fakeRes, () => {});
    return headerValue;
}

const GEOSPATIAL_CACHE_CONTROL = cacheControlFor(300, 600);
```
This helper instantiates a minimal mock response object (`fakeRes`) and executes the middleware against it to capture the exact header string generated by our system. This value is then assigned to `GEOSPATIAL_CACHE_CONTROL` for our test assertions.

## Technical Decisions

### Centralization over Redundancy
We chose to rely entirely on Express middleware for header injection. Having both middleware and inline route-level header setters is prone to race conditions or overriding errors, and it clutters the core business logic of our geospatial queries.

### Dynamic Test Assertions
Instead of mocking the middleware entirely in tests, we chose to execute it against a lightweight fake response object (`as never` casted to bypass strict TypeScript checks for unused properties). This ensures that our integration tests validate the *actual* output format of our middleware, preventing regression if we ever adjust our caching strategy (e.g., changing `s-maxage` or adding `must-revalidate`).

## How To Re-Implement (Contributor Reference)

If you need to apply this pattern to other routes within the SahiDawa API, follow these steps:

1. **Identify Redundant Headers:** Look for manual `res.setHeader("Cache-Control", ...)` calls inside route controllers that already utilize `cacheMiddleware` at the router level.
2. **Clean the Controller:** Remove the local header-setting helpers and constants from the route file. Ensure the router configuration applies the middleware correctly:
   ```typescript
   router.get("/your-endpoint", cacheMiddleware(300, 600), handler);
   ```
3. **Update the Tests:** Avoid hardcoding expected header strings in your test files. Instead, import `cacheMiddleware` and use the dynamic extraction pattern:
   ```typescript
   function cacheControlFor(durationSeconds: number, staleWhileRevalidateSeconds: number): string {
       let headerValue = "";
       const fakeRes = { setHeader: (name: string, value: string) => (headerValue = value) } as never;
       cacheMiddleware(durationSeconds, staleWhileRevalidateSeconds)({} as never, fakeRes, () => {});
       return headerValue;
   }
   ```
4. **Assert Dynamically:** Use the returned value of `cacheControlFor(duration, stale)` in your `supertest` assertions:
   ```typescript
   expect(res.headers["cache-control"]).toBe(GEOSPATIAL_CACHE_CONTROL);
   ```

## Impact on System Architecture

This refactoring improves the maintainability of our backend API by establishing a single source of truth for HTTP caching headers. By decoupling the route handlers from specific header strings, we make it trivial to adjust caching policies globally across SahiDawa. For example, if we need to adjust the `stale-while-revalidate` window for rural health clinics or pharmacies to account for network latency, we only need to modify the parameters passed to `cacheMiddleware` at the router level or the middleware implementation itself, without touching the underlying route controllers or hardcoded test strings.
